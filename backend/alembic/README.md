# Database Migrations with Alembic

This directory contains database migration scripts managed by Alembic.

## Overview

Alembic is a database migration tool for SQLAlchemy. It allows you to:
- Track database schema changes over time
- Apply migrations to upgrade the database
- Rollback migrations to downgrade the database
- Generate migrations automatically from model changes

## Directory Structure

```
backend/alembic/
├── versions/          # Migration scripts
├── env.py            # Migration environment configuration
├── script.py.mako    # Template for new migrations
└── README.md         # This file
```

## Prerequisites

1. **Database Running**: Ensure PostgreSQL is running
   ```bash
   docker compose up -d postgres
   ```

2. **Environment Variables**: Set DATABASE_URL if different from default
   ```bash
   export DATABASE_URL="postgresql://meduser:medpass_dev@localhost:5432/medical_diagnosis"
   ```

## Common Commands

### Using the Helper Script (Recommended)

```bash
# Apply all pending migrations
python backend/migrate.py upgrade

# Rollback one migration
python backend/migrate.py downgrade

# Rollback all migrations
python backend/migrate.py downgrade-base

# Show current migration version
python backend/migrate.py current

# Show migration history
python backend/migrate.py history

# Create a new migration
python backend/migrate.py create "Add new column to users table"
```

### Using Alembic Directly

```bash
# Apply all pending migrations
python -m alembic upgrade head

# Rollback one migration
python -m alembic downgrade -1

# Rollback all migrations
python -m alembic downgrade base

# Show current migration version
python -m alembic current

# Show migration history
python -m alembic history

# Create a new migration (autogenerate from models)
python -m alembic revision --autogenerate -m "Add new column to users table"

# Create a blank migration (manual)
python -m alembic revision -m "Custom migration"
```

## Initial Migration

The initial migration (`20260228031116_initial_migration_with_all_models.py`) creates all database tables:

- **users**: User accounts with authentication
- **user_profiles**: User profile information and preferences
- **diseases**: Disease information and metadata
- **symptoms**: Symptom catalog with synonyms
- **disease_translations**: Multi-language disease information
- **analyses**: Symptom analysis records
- **reports**: Generated PDF reports

### Tables Created

1. **users**
   - User authentication and account management
   - Fields: id, email, password_hash, created_at, updated_at, is_active, is_verified, role

2. **user_profiles**
   - User demographics and preferences
   - Fields: id, user_id, age, gender, medical_history, emergency_contact, language_preference, theme_preference

3. **diseases**
   - Disease information database
   - Fields: id, name, description, severity, common_symptoms, specialist, precautions, lifestyle_recommendations, urgency_guidelines

4. **symptoms**
   - Symptom catalog with synonyms
   - Fields: id, canonical_name, category, synonyms, related_symptoms

5. **disease_translations**
   - Multi-language disease information
   - Fields: id, disease_id, language_code, name, description, precautions, lifestyle_recommendations, urgency_guidelines

6. **analyses**
   - Symptom analysis records
   - Fields: id, user_id, analysis_id, symptoms, predictions, risk_level, recommendations, created_at, notes

7. **reports**
   - Generated PDF reports
   - Fields: id, report_id, analysis_id, format, file_path, file_size, created_at, expires_at

## Testing Migrations

### Test Migration Up

```bash
# Apply all migrations
python backend/migrate.py upgrade

# Verify tables were created
psql -U meduser -d medical_diagnosis -c "\dt"
```

### Test Migration Down

```bash
# Rollback all migrations
python backend/migrate.py downgrade-base

# Verify tables were dropped
psql -U meduser -d medical_diagnosis -c "\dt"
```

### Test Migration Up Again

```bash
# Apply migrations again
python backend/migrate.py upgrade

# Verify tables were recreated
psql -U meduser -d medical_diagnosis -c "\dt"
```

## Creating New Migrations

When you modify database models:

1. **Update the model** in `backend/models/`
2. **Generate migration**:
   ```bash
   python backend/migrate.py create "Description of changes"
   ```
3. **Review the generated migration** in `backend/alembic/versions/`
4. **Test the migration**:
   ```bash
   python backend/migrate.py upgrade
   ```
5. **Test rollback**:
   ```bash
   python backend/migrate.py downgrade
   python backend/migrate.py upgrade
   ```

## Best Practices

1. **Always review autogenerated migrations** - Alembic may not detect all changes correctly
2. **Test migrations in development** before applying to production
3. **Never modify applied migrations** - Create a new migration instead
4. **Keep migrations small and focused** - One logical change per migration
5. **Add data migrations separately** - Don't mix schema and data changes
6. **Backup production database** before applying migrations

## Troubleshooting

### Migration fails with "relation already exists"

The table already exists in the database. Either:
- Drop the table manually
- Rollback to a previous migration
- Mark the migration as applied: `python -m alembic stamp head`

### Migration fails with "column does not exist"

The migration is trying to modify a column that doesn't exist. Check:
- Current database state
- Migration order
- Previous migrations

### Can't connect to database

Check:
- Database is running: `docker compose ps`
- DATABASE_URL is correct
- Database credentials are correct

### Autogenerate doesn't detect changes

Ensure:
- Models are imported in `backend/models/__init__.py`
- Models inherit from `db.Model`
- Changes are actually different from current schema

The database URL is read from:

2. Flask app configuration
